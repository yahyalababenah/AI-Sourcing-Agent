### Bug 2 — Thread-unsafe lazy engine initialisation in `_get_engine()`

| Field | Value |
|-------|-------|
| **File** | [`app/modules/documents/ocr_client.py`](../app/modules/documents/ocr_client.py:88) |
| **Area** | Document Pipeline — OCR engine initialisation |
| **Type** | Concurrency — Race condition |
| **Severity** | 🟡 Medium |
| **Status** | ✅ **Fixed** |

**The Bug:**
The global `_engine` variable was initialised lazily in `_get_engine()` without any thread lock. If two Celery workers executed tasks simultaneously, both could pass the `if _engine is None` check and initialise the engine concurrently.

```python
_engine: PPStructure | None = None

def _get_engine() -> PPStructure:
    global _engine
    if _engine is None:  # ⚠️ Not thread-safe
        _engine = PPStructure(...)  # Both workers may reach here
    return _engine
```

**Impact:**
Potential double initialisation of PaddleOCR engine (memory waste), or rare race-condition crashes if both workers try to write to `_engine` simultaneously.

**Fix Applied:**

Added **double-checked locking** pattern using `threading.Lock()` in [`app/modules/documents/ocr_client.py`](../app/modules/documents/ocr_client.py):

1. Added `import threading` at the top of the file (line 28)
2. Created module-level lock: `_engine_lock = threading.Lock()` (line 87)
3. Wrapped the initialisation block inside `with _engine_lock:` with a **second check** after acquiring the lock (lines 100-124)

```python
_table_engine = None
_engine_lock = threading.Lock()

def _get_engine():
    global _table_engine
    if _table_engine is None:
        with _engine_lock:
            # Double-check after acquiring lock — another thread may have
            # already initialised the engine while we were waiting.
            if _table_engine is not None:
                return _table_engine
            # ... initialise engine ...
    return _table_engine
```

**Why this is safe:**
- The **first `if _table_engine is None`** is lock-free — fast path once engine is initialised (no lock contention)
- The **`with _engine_lock:`** ensures only one thread enters the critical section
- The **second `if _table_engine is not None`** prevents double initialisation if another thread already set it while waiting
- `threading.Lock()` is a **reentrant** lock from stdlib — no extra dependencies

**Verification:**
- ✅ All 13 unit tests pass (TestRepairVisionJson, TestValidateResult)
- ✅ Imports resolve correctly: `_get_engine`, `_engine_lock` loadable
- ✅ `_engine_lock` is a proper `threading.Lock()` instance

### Bug 7 — `_extract_categories_from_db_products()` uses substring matching (too loose)

| Field | Value |
|-------|-------|
| **File** | [`app/modules/intake/matcher.py`](../app/modules/intake/matcher.py:164) |
| **Area** | RFQ Matching — Category extraction |
| **Type** | Functional Error — False positive categories |
| **Severity** | 🟠 High |
| **Status** | ✅ **Fixed** |

**The Bug:**
The keyword matching used `if keyword in combined` (substring match), not word-boundary matching. This meant:

```python
# Line 164-166: Substring match — too loose
for keyword, category in keyword_map.items():
    if keyword in combined:  # ⚠️ "rice" matches "price", "oil" matches "soiled"
        categories.add(category)
```

**Concrete examples of false positives:**
- Product named "price list" → matches keyword `rice` → category `food & beverage` ❌
- Product "soiled fabrics" → matches keyword `oil` → category `personal care` ❌
- Product "machinery parts" → matches keyword `parts` → category `machinery` ✅ (correct, but fragile)
- Product "tool box" → matches `tool` AND `box` → `machinery` AND `paper & packaging` ❌

**Impact:**
RFQs get assigned to wrong categories during matching, leading to:
- Suppliers matched to irrelevant RFQs (wasted supplier attention)
- Suppliers missing relevant RFQs because _other_ suppliers with wrong matches were ranked higher
- Inflated match scores from spurious category overlaps

**Fix Applied:**
Changed the matching logic at [`matcher.py:168`](../app/modules/intake/matcher.py:168) to use word-boundary regex:

```python
for keyword, category in keyword_map.items():
    # Use word-boundary matching to avoid false positives like
    # "rice" matching inside "price" or "oil" inside "soil".
    if re.search(rf"\b{re.escape(keyword)}\b", combined):
        categories.add(category)
```

- `\b` ensures the keyword matches only as a whole word (word boundary on both sides)
- `re.escape()` prevents regex injection from special characters in keywords
- Added `import re` at the top of the file

**Verification:**
- All 27 intake API tests pass (✅ `tests/test_intake/test_intake_api.py`)
- Keyword "rice" no longer matches "price list"
- Keyword "oil" no longer matches "soiled fabrics"
- All legitimate matches (e.g., "rice" in "long grain rice") still work correctly

### Bug 20 — `intakeService.updateStatus()` sends wrong request format (resolved as query param)

| Field | Value |
|-------|-------|
| **File** | [`frontend/src/services/intakeService.ts:31`](../frontend/src/services/intakeService.ts:31) |

| **Type** | Functional Error — Wrong request format |
| **Severity** | 🟠 High |

**The Bug (original):**
```typescript
updateStatus: (id: string, status: string) =>
    api.put<RFQ>(API.INTAKE.RFQ_STATUS(id), status, {  // ⚠️ Sends raw string
        headers: { "Content-Type": "application/json" },
    }).then((r) => r.data),
```

The original code sent the raw `status` string as the PUT request body with `Content-Type: application/json`. However, the backend [`update_rfq_status_endpoint`](../app/modules/intake/router.py:220) declares `new_status: RFQStatus` as an undecorated parameter, which FastAPI treats as a **query parameter** (since `RFQStatus` is a `str` enum, not a Pydantic model). **Impact:**
Status updates from the frontend would fail — either as 422 Validation Error (if sent as raw string body) or field mismatch (if sent as JSON body with wrong key).

**✅ Fix Applied:**
Changed from sending raw string as body to using axios `params` config for query parameter:
```typescript

    api.put<RFQ>(API.INTAKE.RFQ_STATUS(id), null,
        { params: { new_status: status } }
    ).then((r) => r.data),
```

This matches the backend's expectation of `?new_status=processing` as a query parameter.
