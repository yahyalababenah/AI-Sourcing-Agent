# حالة مشروع AI-Sourcing Hub — تقرير شامل (باك إند + فرونت إند)

> آخر تحديث: 2026-07-07 · الفرع الحالي: `fix/performance-optimization` · آخر tag: `v1-design-migration`
> هذا التقرير مبني على **فحص الكود الفعلي** (راوترات، موديلات، migrations، اختبارات، صفحات) لا على وثائق قديمة قد لا تكون دقيقة. للتفاصيل التنفيذية الكاملة لهجرة الواجهة راجع `PLAN.md`، وللقواعد الثابتة راجع `CLAUDE.md`.

---

## 1. نظرة عامة

**AI-Sourcing Hub** منصة B2B **عامة متعددة الفئات** (وليست مقتصرة على قطاع
الإضاءة الصناعية كما ورد بوثائق أقدم — تم التأكد من الكود أن أي تحديد
نطاق كهذا أُزيل بالكامل) تربط المستوردين الأردنيين بالموردين/المصانع
الصينية. محرك المطابقة (`app/modules/intake/matcher.py`) يدعم حالياً
**10 فئات منتجات عامة**: العناية الشخصية، الأغذية والمشروبات، المنسوجات،
الإلكترونيات، الآلات، المعادن والأدوات، الأثاث، التغليف، الكيماويات،
الزجاج والسيراميك.

**القيمة الأساسية**: العميل يرى التكلفة الكاملة المقدّرة (سعر أساسي +
شحن + جمارك + ضريبة) لأي منتج قبل التواصل مع مندوب، ثم يطلب عرض سعر
رسمي يمر عبر محرك مطابقة تلقائي مع الموردين.

**البنية التقنية**:

| الطبقة | التقنية |
|---|---|
| Backend | Python 3.13 + FastAPI (Modular Monolith، async) |
| Frontend | React 18 + TypeScript + Vite (واجهة عربية RTL) |
| قاعدة البيانات | PostgreSQL + SQLAlchemy (async) + asyncpg |
| الترحيلات | Alembic — 18 ملف migration |
| كاش/طابور | Redis |
| مهام غير متزامنة | Celery + Celery Beat |
| تخزين ملفات | MinIO (متوافق S3) |
| OCR/AI استخراج | PaddleOCR PP-Structure محلي + LLM (DeepSeek أساسي، OpenRouter/Together fallback) |
| مراقبة | Prometheus + Sentry + Grafana + Flower |
| نشر | Docker/docker-compose + Railway + Hugging Face Spaces (entrypoint مخصص) |

## 3. الآليات الجوهرية المؤكَّدة فعلياً (ليست ادّعاءً تسويقياً)

1. **محرك مطابقة RFQ حصري ثم عام**: يستخرج فئة المنتج من الكيانات
   المستخرجة بالـ AI، يطابقها بالموردين (تطابق كتالوج مباشر أو تداخل
   تخصص)، يمنحهم نافذة حصرية **3 ساعات** (فهرس `ix_rfq_is_public_exclusive_deadline`
   مخصص لذلك)، ثم `is_public=true` تلقائياً إن لم يُستجَب.
2. **خط أنابيب OCR/AI حقيقي متعدد المراحل** (`documents/ocr_client.py`):
   - Phase 0 (بلا LLM): Excel/CSV/Word عبر pandas، جداول PDF عبر pdfplumber.
   - Phase 1 (fallback): PDF نصي عبر pypdfium2، صور/ممسوح ضوئياً عبر
     **PaddleOCR PP-Structure محلي بالكامل** (يتدهور بلطف إن تعطّلت البيئة).
   - Phase 2: نص خام → LLM لتحويله JSON مهيكل (مُجزّأ للكتالوجات الطويلة)
     + `json_repair.py` لإصلاح JSON مشوّه من الـLLM.
   - مهمة Celery: رفع → MinIO → `processing` → استخراج → `extracted_entities`
     JSONB → `extracted`/`failed`.
   - `catalog/service.py` يزامن المستند المستخرَج → صفوف `CatalogProduct`
     ببوابة `ProductReviewStatus` (اعتماد بشري قبل الظهور بالسوق العام —
     **Human-in-the-loop حقيقي**، ليس مجرد مفهوم بالتوثيق).
3. **مراجعة بشرية قبل النشر**: عمود `review_status` (pending/approved/
   rejected) — البحث العام يعرض `approved` فقط. المندوب يراجع من صفحة
   مخصصة، والأدمن له طبقة إشراف كاملة فوق ذلك (T9.4.1 بالفرونت إند).
4. **إشعارات فورية حقيقية عبر SSE**: الباك إند يبث أحداثاً حقيقية
   (`new_rfq`, `quote_ready`) والفرونت إند (`useNotifications.ts`) يفتح
   `EventSource` حقيقياً يستهلكها — ليست أيقونة واجهة فارغة.
5. **Token blacklist مزدوج عند تسجيل الخروج**: refresh token يُبطَل عبر
   Redis blacklist بالـjti، والaccess token يُبطَل بمقارنة توقيت إصداره
   بتوقيت "آخر تسجيل خروج" المحفوظ بـRedis.
6. **لا كود ناقص/placeholder بالباك إند**: فحص شامل عن TODO/FIXME/
   NotImplementedError/mock لم يُظهر شيئاً جوهرياً — الاستثناء الوحيد
   المؤكَّد هو `GET /pricing/rules/{id}/history` (سجل تدقيق التسعير)
   مُرجَع فارغاً صراحةً، مؤجَّل لجدول audit-log مستقبلي (Phase 5).

### 3.1 محرك التسعير (`pricing/engine.py`, 761 سطر) — الأكثر نضجاً

`PricingEngine`/`PricingContext` ينفّذان حساب التكلفة الواصلة الكاملة:
تحويل يوان→دولار→عملة محلية، تقدير حجم CBM من الوزن، شحن/وحدة، جمارك %،
رسوم تخليص ثابتة، عمولة على التكلفة التراكمية، خصومات MOQ/دفع مبكر،
ضريبة، هامش مستهدف. يدعم `CustomRule` (نسبة/ثابت/معادلة) عبر
`formula.py` (مُقيِّم معادلات آمن) وجداول رسوم HS Code حقيقية (migrations
015، 016).

## 4. قاعدة البيانات

**18 migration** بـAlembic تغطي دورة بناء تدريجية: مخطط أولي → ربط
RFQ-عميل → مرونة agent → خصومات فئات التسعير → ملفات مستخدمين شخصية →
منتجات كتالوج → حالة تتبّع الشحنة → حالة توثيق المورد → جداول الشات →
جداول مطابقة RFQ → فهرس مهلة الحصرية → حالة مراجعة الكتالوج → مرونة FK
منتج ببنود عرض السعر → FK منتج كتالوج ببنود عرض السعر → جداول رسوم HS →
معادلات تسعير + ضريبة. تغطية دومين كاملة: مستخدمون/ملفات، RFQs، منتجات
كتالوج، عروض أسعار/بنودها، شات، مطابقة RFQ، تتبّع شحنة، رموز HS، قواعد
تسعير.

## 5. الاختبارات

**~391 اختبار عبر 41 ملفاً** بالباك إند (`tests/`: test_auth، test_documents،
test_intake، test_output، test_pricing، unit/، security/، integration/) —
تغطية جادة وليست سطحية: أثقل تغطية بمحرك التسعير (39+22+10 اختبار)،
المستندات (28)، الاستقبال/المطابقة (27)، المصادقة (22). اختبارات
`security/` تغطي CSP headers، صلاحية URL موقّعة لـMinIO، حدود RBAC، حقن
SQL.

بالفرونت إند: **257 اختبار Vitest ناجح** (وقت آخر تحديث لـ`PLAN.md`)
تغطي كل شاشة مُهاجَرة بنمط Desktop/Mobile — تحقّق فعلي لا نظري لكل مهمة.

CI (`.github/workflows/ci.yml`) بثلاث مهام حقيقية: `backend-unit` (pytest،
عند كل push)، `frontend-unit` (Vitest، عند كل push)، `integration-security`
(بخدمة Postgres 16، عند كل PR).

## 6. من المستفيدون (الأدوار الثلاثة)

| الدور | من هو | ماذا يفعل بالمنصة | الألوان |
|---|---|---|---|
| **المندوب/المورد (Agent)** | يمثّل مصنعاً صينياً | يرفع كتالوج المنتجات، يدير طلبات RFQ الواردة (حصرية/عامة)، يبني عروض أسعار بمحرك تسعير حقيقي، ينشر ريلز تسويقية | `supplier-*` (زمردي) |
| **المستورد (Client)** | تاجر/شركة أردنية | يتصفح السوق العالمي، يشاهد ريلز المصانع، يطلب عروض أسعار، يتابع شحناته، يتواصل مع الموردين | `importer-*` (أزرق بحري) |
| **الأدمن (Admin)** | فريق المنصة | يراقب صحة النظام، يعتمد/يرفض توثيق الموردين، يراجع الكتالوج العالمي المُستخرج بالذكاء الاصطناعي، يدير قواعد التسعير/رسوم HS/المستخدمين | slate محايد |

## 7. دورة حياة المستورد (Client Journey) — من الباك إند للفرونت إند

```
تسجيل الدخول (JWT حقيقي، تبويب "مستورد")
        ↓
لوحة التحكم: 4 مؤشرات حقيقية مشتقّة من RFQs/عروض فعلية + "جديد اليوم
من المصانع" (كتالوج حقيقي، بوابة review_status=approved فقط)
        ↓
اكتشاف: السوق العالمي (بحث نصي كامل GIN tsvector) أو ريلز
        ↓
طلب عرض سعر جديد → استخراج AI (عربي→صيني) → محرك المطابقة يوزّعه
تلقائياً على الموردين المناسبين (نافذة حصرية 3 ساعات ثم عام)
        ↓
إشعار SSE فوري عند وصول عرض سعر حقيقي
        ↓
محادثة حية (SSE) لتفاوض التفاصيل
        ↓
قبول العرض → توليد PDF (Celery) → تتبّع الشحنة (خط زمني حقيقي)
        ↓
صفقة مكتملة → تنعكس بمؤشرات لوحة التحكم + الملف الشخصي
```

## 8. دورة حياة المندوب (Agent Journey)

```
تسجيل الدخول → لوحة التحكم (شريط "تنتظر ردّك" + Kanban)
        ↓
رفع كتالوج → خط أنابيب OCR/AI (PaddleOCR محلي + LLM هيكلة) →
"قيد المراجعة" → موافقة المندوب ثم اعتماد الأدمن (طبقة إشراف مزدوجة)
        ↓
صندوق الوارد: مباريات حصرية (عدّاد تنازلي 3 ساعات) + سوق عام
        ↓
"قدّم عرضاً" → بانٍ عروض متصل بمحرك التسعير الحقيقي (HS codes،
جمارك، ضريبة، عمولة، خصومات) → إرسال → إشعار SSE فوري للعميل
        ↓
قبول العميل → تحديث مراحل تتبّع الشحنة يدوياً
        ↓
بالتوازي: نشر ريلز (لوحة أداء حالياً 0/— صادقة — لا backend
لتحليلات الريلز بعد، هذه أكبر فجوة بيانات بالمنصة)
```

**نقاط التقاطع**: نفس RFQ يُرى فوراً من الطرفين، عرض السعر يظهر لحظياً
عند العميل، غرفة الشات مشتركة بفقاعات ملوّنة حسب الدور مع زر "أرسل عرض
سعر" للمندوب فقط، خط تتبّع الشحنة واحد يراه الطرفان، والكتالوج المعتمد
من الأدمن هو ما يظهر بالسوق العالمي للطرفين.

## 9. الهوية البصرية (Design System)

- **RTL أساسي** — `text-start/end` لا `text-right/left` ثابتة.
- ثلاث عائلات ألوان فقط من `lib/tokens.ts`: `supplier-*` (مندوب)،
  `importer-*` (مستورد، أزرق بحري)، `brand-*` (مشترك) — **`#4F46E5`
  (indigo) محظور نهائياً**، رُصدت وأُصلحت 3 تسريبات فعلية له.
- **الأرقام المالية**: slate بحت + `tabular-nums`، الأخضر للإجمالي
  النهائي فقط.
- **نمط الملفين إلزامي**: كل شاشة رئيسية = ملف Desktop + ملف Mobile
  منفصلين + مبدّل `useMediaQuery` — مطبَّق على كل الشاشات الأساسية.
- **مبدأ الصدق بالبيانات** (متكرر بكل مرحلة هجرة): أي رقم/شارة بلا
  backend حقيقي (تحليلات ريلز، تقييمات، شهادات) يُعرض صفر/شرطة/تنويه
  صريح بدل تلفيق أرقام.

## 10. حالة الفرونت إند — بالمرحلة (من `PLAN.md`، 12 مرحلة)

- المرحلة 0-1: الأساس + المكوّنات المشتركة (Sidebar/Drawer/BottomNav/TopBar/ذرّيات/حركة)

- المرحلة 6: الريلز (أستوديو المورد + مشغّل المستورد)
- المرحلة 7: الملفات الشخصية (مستورد + مورد)
- المرحلة 8: كل صفحات سير العمل (RFQ، صندوق الوارد، عروض الأسعار، طلباتي، السوق العالمي، المحادثات، تتبّع الشحنات)
- المرحلة 9: كل صفحات الأدمن (لوحة التحكم، مراقبة النظام، إدارة المستخدمين، قواعد التسعير، جداول HS، الكتالوج العالمي بمنطق مراجعة/اعتماد)
- المرحلة 10: ربط API شامل + حالات تحميل/فراغ/خطأ+إعادة محاولة

**مكتمل جزئياً 🟡**:
- المرحلة 11: جرس الإشعارات ✅ + مبدّل اللغة ✅، لكن i18n كامل (~163 ملفاً بنص عربي غير مترجَم) وتدقيق لمسي شامل (44px لكل زر) لم يُنفَّذا بعد

## 11. صفحة الهبوط (Landing Page) — جديد، تصميم فقط

تم تصميم صفحة هبوط عامة (قبل تسجيل الدخول) — الفكرة والبنية أصبحت واضحة
ومعمولة، لكنها **تصميم بصري منفصل حتى الآن، غير مربوط بعد بكود
`frontend/src`** ولا موجودة كراوت فعلي بالمنصة. الأقسام المصمَّمة:

1. Hero — "السوق الصيني بعين عربي" + عمود ثقة + CTAs مزدوجة (مستورد/مندوب) + مبدّل لغة AR/EN/中文
2. مقارنة "لماذا هذه المنصة" مقابل "الاستيراد اليوم"
3. 3 خطوات: اكتشف → اعرف تكلفتك → اطلب عرض سعر
4. عمودان "لمن المنصة" بألوان `supplier-*`/`importer-*` المعتمدة
5. شبكة ثقة (شفافية التكلفة، محتوى حقيقي، موردون موثّقون، Escrow)
6. لقطات ريلز حقيقية بعدّاد RFQ (نفس منطق `ReelTile`)
7. CTA ختامي + نموذج تسجيل مبكّر

**الخطوة التالية لها**: تحويلها لصفحة React فعلية (Desktop + Mobile) وربطها براوت عام `/` قبل شاشة تسجيل الدخول.

## 12. أهم الفجوات الحقيقية المتبقية (مؤكَّدة من الكود، بترتيب الأهمية)

1. **لا FK مباشر بين RFQ ومنتج الكتالوج** — جدول `rfqs` لا يحمل
   `catalog_product_id`؛ العلاقة تمر عبر `Product` (كيانات مستخرَجة من
   طلب العميل) لا `CatalogProduct` مباشرة — يصعب على المندوب لاحقاً ربط
   الطلب 100% بعنصر كتالوج محدد.
2. **لا backend لتحليلات الريلز** — كل عدّادات RFQ/مشاهدات/معدل تحويل
   بأستوديو الريلز صفر/شرطة صادقة، ليست مقاسة فعلياً بعد.
3. **صفحة الإعدادات (Settings) شبه فارغة** — بلا ربط بيانات حقيقي.
4. **الانتقال التلقائي حصري→عام بعد 3 ساعات** — الموديل والفهرس جاهزان
   لكن لم يُتأكَّد قطعياً من وجود Celery Beat task ينفّذه تلقائياً.
5. **i18n كامل** (~163 ملفاً) وتدقيق لمسي شامل — مؤجَّلان لجلسات مخصّصة.
6. **صفحة الهبوط** — تصميم فقط، غير مربوطة بالكود بعد.
7. سجل تدقيق قواعد التسعير (`/pricing/rules/{id}/history`) — endpoint موجود لكن يعيد قائمة فارغة صراحةً (مؤجَّل لجدول audit-log مستقبلي).

المنصة **منتج B2B عام متعدد الفئات يعمل فعلياً من طرف لطرف**: رفع كتالوج
→ استخراج OCR/AI حقيقي → مراجعة بشرية مزدوجة (مندوب ثم أدمن) → ظهور
بالسوق العام → طلب عرض سعر → مطابقة تلقائية حصرية ثم عامة → تسعير حقيقي
بمحرك HS/جمارك/ضريبة → توليد PDF → إشعار SSE فوري → قبول → تتبّع شحنة.
الباك إند ناضج وموثَّق بالاختبارات (391 اختبار)، والفرونت إند أعاد بناء
كل الشاشات الأساسية على نظام تصميم موحّد (257 اختبار). العمل المتبقي
الأبرز: صفحة الهبوط العامة (تصميم جاهز، تحويل لكود متبقٍّ)، ربط RFQ
بالكتالوج مباشرة، i18n كامل، وهجرة ~9 صفحات وظيفية متبقية لنمط التصميم
